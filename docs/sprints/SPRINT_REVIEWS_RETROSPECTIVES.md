# KERNO — Sprint Reviews and Retrospectives

## Objective

This document records sprint review and retrospective notes for the Stage 4 development phase.

## Sprint 1

### Delivered

- Repository initialized.
- Initial frontend/backend organization prepared.
- Documentation structure started.
- GitHub workflow and issues prepared.

### Retrospective

| Question | Answer |
|---|---|
| What worked well? | The project scope and repository structure were clarified early. |
| What was difficult? | Aligning the MVP scope with portfolio deliverables. |
| Improvement | Keep issues small and linked to clear user stories. |

## Sprint 2

### Delivered

- Backend modules structured around auth, users, suppliers, stores, products, categories and requests.
- Prisma schema and migrations created.
- PostgreSQL persistence connected through Prisma.
- Initial API documentation added.

### Retrospective

| Question | Answer |
|---|---|
| What worked well? | The modular monolith kept the backend readable. |
| What was difficult? | Keeping the API complete while avoiding scope drift. |
| Improvement | Document endpoints and expected payloads as routes are created. |

## Sprint 3

### Delivered

- Main frontend pages implemented.
- Landing, login and registration pages prepared.
- Store and supplier dashboards added.
- Catalog, product detail, supplier detail and request pages added.

### Retrospective

| Question | Answer |
|---|---|
| What worked well? | The MVP user journey became visible and demonstrable. |
| What was difficult? | Maintaining visual consistency across many pages. |
| Improvement | Validate visual consistency page by page before merging. |

## Sprint 4

### Delivered

- Frontend visual harmonization improved.
- Dashboard, catalog and request pages polished.
- Lint and build checks validated.
- Backend syntax checks validated.

### Retrospective

| Question | Answer |
|---|---|
| What worked well? | The project stayed focused on QA and readiness. |
| What was difficult? | Avoiding unnecessary changes while improving quality. |
| Improvement | Keep written test evidence after validation passes. |

## Sprint 5

### Delivered

- Final Stage 4 checklist prepared.
- Demo scenario available.
- Manual review preparation documented.
- Links to repository, project board, issues, branches and test evidence centralized.

### Retrospective

| Question | Answer |
|---|---|
| What worked well? | The MVP is structured, documented and aligned with scope. |
| What was difficult? | Proving the work clearly with documentation and tests. |
| Improvement | Keep oral review notes concise and technical. |
